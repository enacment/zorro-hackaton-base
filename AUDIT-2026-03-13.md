## Pass 1 — Análisis Estático

### ✅ Lo que está bien

**Estructura general:** Limpia y bien dividida:
```
apps/ → código por equipo (+ template base)
context/ → PRD + memoria (fuente de verdad AI)
docs/ → guardrails, arquitectura, continuidad
playbooks/ → Codex + Antigravity instrucciones
prd/ → PRD por equipo (mirror de context/)
scripts/ → checklist de bootstrap
teams/ → info por equipo
```

**AGENTS.md:** Muy bien escrito. Define:
- Flujo obligatorio de 6 pasos para el agente
- Restricciones no negociables
- Formato de salida esperado
- Alineado 100% con el prompt del taller (context/PRD.md como fuente)

**Auth base (Template):** Código C# limpio, desacoplado, listo para usar:
- `IAuthProvider` → interfaz
- `MockAuthProvider` → funcional para DEV (usuario/contraseña cualquiera)
- `LegacyDllAuthProvider` → placeholder para DLL real
- `HealthCheckService` → ping a API endpoint

**CI Workflow:** `prd-reference-check.yml` valida que todo PR referencie `context/PRD.md`. Bien pensado.

**Seguridad:** Sin secretos hardcodeados. `appsettings.template.json` usa `./local-secrets/` para DLL paths — correcto.

---

### ⚠️ Observaciones

**1. Los repos de equipo no tienen código fuente**
```
apps/Zorro.Planogramas.App/  → solo README.md
apps/Zorro.Traspasos.App/    → solo README.md
apps/Zorro.Precios.App/      → solo README.md
```
Los equipos deben copiar o basarse en `Zorro.Template.App/src/` para arrancar. Si no saben esto, perderán tiempo en el Kickoff.

**Recomendación:** Agregar una línea en cada README de equipo:
> _"Punto de partida: copiar `apps/Zorro.Template.App/src/` a `apps/Zorro.{Equipo}.App/src/` como base."_

**2. No hay `.csproj` ni `.sln`**
El código C# existe pero sin proyecto MAUI compilable. Los equipos no pueden hacer `dotnet build` al arrancar.
- Riesgo: el sprint 1 se puede bloquear si el equipo no sabe crear un proyecto MAUI desde cero.
- El README menciona "Ejecutar build baseline" en el bootstrap-checklist pero no hay proyecto para compilar.

**Recomendación:** O se genera el scaffold MAUI antes del taller, o se aclara explícitamente que el código en `src/` son _archivos de referencia_, no un proyecto compilable directamente.

**3. PRDs todos vacíos**
```
context/PRD.md       → "Pendiente de carga"
prd/planogramas/PRD.md → "Pendiente de carga"
```
`context/PRD.md` es la fuente de verdad del agente según AGENTS.md. Si está vacío, Codex no tiene contexto funcional.

**Acción requerida hoy:** Cada equipo debe llenar su PRD antes o durante el Kickoff. Sin PRD, el agente opera sin referencia.

## Pass 2 — Verificación Runtime

**Dependencias:**
```
No hay package.json, Gemfile, requirements.txt, ni .csproj
→ No se puede ejecutar npm install / dotnet build
→ El repo es scaffolding + docs + C# snippets, no un proyecto compilable en este estado
```

**CI Workflow syntax:**
```yaml
uses: actions/github-script@v7  ✅ versión válida
trigger: pull_request (opened, edited, synchronize, reopened) ✅
lógica: verifica body del PR contiene "context/prd.md" ✅
```
Sin errores de sintaxis. Funciona cuando se haga el primer PR.

**`.gitignore`:** No está incluido en el repo. Riesgo: un equipo accidentalmente podría commitear `appsettings.json` con credenciales reales o los DLLs.

### Fix 2 — Aclarar en READMEs de equipo el punto de arranque (5 min)

## Arranque de código
Copia el contenido de `apps/Zorro.Template.App/src/` a `apps/Zorro.{Equipo}.App/src/` como punto de partida.

### Fix 3 — PRDs (crítico para hoy)
Cada equipo debe tener su PRD en `context/PRD.md` (o el equipo puede usar `prd/{equipo}/PRD.md` y copiar al context).
**Sin PRD cargado, Codex no tiene fuente de verdad y el prompt base del taller no funciona correctamente.**

## Veredicto final

**El repo está bien diseñado y documentado.** La arquitectura de contexto (AGENTS.md + context/ + prd/) es sólida y alineada con el flujo del taller.

Los dos bloqueantes para el sprint 1 de hoy:

Ninguno de los dos bloquea el taller si se comunica en el Kickoff (9:00–9:20). Con 20 minutos de setup y PRD basic cargado, los equipos pueden arrancar Sprint 1.
